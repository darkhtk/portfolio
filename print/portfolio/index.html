DeskRelay first

Flagship remote AI product and its kernel support

AI Product Platform

DeskRelay

The flagship project: a browser/mobile control plane for driving local agent execution on a user's own paired PC. The product surface coordinates identity, pairing, relay, device diagnostics, and UX while the connector owns local execution.

  • Problem: Local agent tools are powerful because they sit beside the user's repos, shell, and tokens, but that also pins work to one keyboard.
  • Build: SolidJS frontend, Cloudflare Workers backend, Durable Object relay, Bun PC connector, device lifecycle, diagnostics, Android packaging, and release documentation.
  • Outcome: A production-shaped alpha and public repository at github.com/darkhtk/deskrelay, demonstrating AI workflow as a real remote product surface.
Agent Workflow Kernel

Ops-Cure Kernel

The supporting kernel work behind the DeskRelay direction: a channel-native state/event model built around Space, Actor, Event, and Behavior.

  • Problem: Remote agent products need canonical state and behavior boundaries, or every new control surface becomes a fork.
  • Build: Generic kernel vocabulary, Discord transport, behavior plugin registry, canonical DB, orchestration behavior, remote_codex behavior, and chat behavior.
  • Outcome: A reusable mental model for DeskRelay-style runtimes: product shell outside, local execution and behavior contracts inside.

AI support around DeskRelay

Local AI runtime thinking and AI experience proof

Local LLM R&D

GenWorld + Ollama

Supporting R&D for the DeskRelay direction: local LLM behavior, fallback rules, memory, and runtime state that can be inspected instead of hidden behind a prompt.

  • Problem: NPC dialogue needed to matter inside gameplay without depending on a fragile cloud-only prompt path.
  • Build: Ollama-backed runtime experiments, memory rules, relationship rules, quest interaction hooks, and fallback-aware data layout.
  • Outcome: A practical R&D bed for local LLM behavior where AI decisions remain visible and tied to game state.
Interactive XR x Generative AI

Neostalgia

Supporting proof that AI needs an operating surface: an XR exhibition system that interviews visitors, generates assets, and stays manageable through telemetry and a dashboard.

  • Problem: Voice input, AI generation, XR sequencing, and exhibition operations had to stay coherent under live visitor traffic.
  • Build: Whisper, GPT, TTS, Skybox AI, Meshy, GCP storage, XR timing, session logging, and an operations dashboard.
  • Outcome: BIFAN XR selection, MSIT Minister Citation, 130+ direct participants, and a public AI XR system that held up in operation.

Unity industrial development

Robotics control and operator tooling

Industrial XR x Robotics

VR Robot Teleoperation

A full Quest 3 to Doosan A0912 control path through Unity, Flask, RoboDK, calibration, safety logic, and later in-house Jacobian IK ownership.

  • Problem: XR input needed to control an industrial robot with stable mapping, safe movement, and operator-readable constraints.
  • Build: Controller filtering, coordinate transforms, bridge APIs, safety boxes, deadman logic, resend control, and IK continuation.
  • Outcome: A public demo and a proof that the AI-centered portfolio still rests on real control, safety, and runtime discipline.
Simulation Tooling / Digital Twin

DXCenter

A Unity authoring environment for LG Energy Solution content with nine EditorWindow tools, mapping rules, and a cleaner runtime path for non-developer operators.

  • Problem: Production content needed repeatable authoring, validation, and handoff without depending on developer-only editing paths.
  • Build: EditorWindow tools, state management, undo, data mapping, preview/runtime separation, and operator-facing authoring flow.
  • Outcome: Lower repeated setup cost and a tooling surface that turns content production into a maintainable workflow.
These Unity industrial systems are not the headline direction. They are the execution base: control, tools, operators, geometry, and delivery habits that make the AI systems credible.

Unity industrial development base

BIM geometry and field delivery habits

These projects are secondary to the AI story, but they matter because they show the Unity industrial development depth behind it: heavy geometry, simulator delivery, hardware setup, and operator-facing runtime work.

BIM Runtime Tooling

WatchBIM

A Unity BIM runtime built around a custom hit engine, station-based cross-section generation, 2D analysis viewer, and a tool-grade camera system for heavy geometry inspection.

  • Problem: Large BIM scenes needed accurate selection, reliable movement, and in-runtime cross-section analysis.
  • Build: Triangle caching, hit filtering, geometry jobs, plane intersection, segment grouping, 3D-to-2D conversion, and slope labeling.
  • Outcome: A practical viewer and analysis tool that goes beyond visualization into geometry and domain logic ownership.
