# PORTFOLIO_REWRITE_PLAN

## 목표 요약

사이트가 최종적으로 다음처럼 읽히도록 재구성한다.

- 산업용 Unity 실시간 시스템을 오래 해온 개발자
- XR, 시뮬레이션, BIM, 에디터 툴링, 자동화, 로봇 연동 같은 어려운 문제를 실제로 풀어온 사람
- 최근에는 AI 협업과 orchestration 방식으로 XR, 에디터, 백엔드, 로봇 제어까지 더 넓게 문제를 푸는 사람
- 중요한 것은 AI 사용 자체가 아니라 실제 동작하는 시스템과 운영 가능한 구조를 완성했다는 점

## 새 정보 구조

### 전체 사이트

1. Home
   산업용 Unity 실시간 시스템 개발자라는 정체성을 먼저 전달하고, AI 활용 방식을 구조적으로 설명하는 메인 허브
2. Projects
   대표 프로젝트 3개를 문제 해결형 케이스 스터디 카드로 보여 주고, 기반 경력 프로젝트를 압축 정리
3. About
   자기 브랜딩보다 일하는 방식, 판단 기준, 협업 철학을 설명
4. Resume
   경력 흐름과 성장 축을 시간순으로 정리
5. Cover Letter
   기존 서사 자산은 유지하되 전체 톤 조정 범위에 따라 후속 정리
6. Case Studies
   상세 증거와 구현 맥락을 제공하는 심화 문서 역할 유지

## 각 페이지의 역할

### Home

- 첫 5초 메시지 전달
- AI 활용 방식을 구조화해 신뢰 형성
- 대표 3개 프로젝트를 선별해 현재 포지셔닝을 증명
- 기반 경력으로 현재 역량의 뿌리를 설명
- 최종 CTA로 연락 동선 정리

### Projects

- 홈에서 강조한 3개 프로젝트를 더 구조적으로 제시
- 나머지 프로젝트는 foundation 묶음으로 압축
- "많이 했다"보다 "어떤 문제를 풀 수 있는 사람인가"가 보이게 구성

### About

- 브랜딩 문구 축소
- 문제 정의, 제약 정리, 병목 해소, 검증 기준, AI 통제 방식 중심 설명

### Resume

- 회사별 나열만이 아니라 성장 단계가 읽히게 재정렬
- 교육용 게임 → VR/AR 안전교육 → BIM/설계 자동화 → 에디터 툴링/디지털트윈 → AI 융합 XR/로봇 제어 흐름 강화

## 홈 섹션 재배치 원칙

홈은 아래 순서를 따른다.

1. Hero
2. How I Actually Use AI
3. Featured AI-Focused Projects
4. Foundation Projects
5. Working Style
6. Final CTA

### Hero 원칙

- 첫 문장은 산업용 Unity 실시간 시스템 개발자
- AI는 최근 확장된 문제 해결 방식으로 뒤따라옴
- CTA는 `대표 프로젝트 보기`, `이력 흐름 보기`처럼 분명한 두 갈래
- 배지, 수식어, 강조 요소는 최소화

### How I Actually Use AI 원칙

- 자랑 섹션이 아니라 통제된 엔지니어링 워크플로우처럼 보이게 설계
- 권장 흐름:
  Problem → Why AI → Application Area → Validation/Control → Outcome
- 또는 역할 구분:
  Human / AI / System / Validation
- 핵심 메시지:
  사람이 구조와 검증을 잡고, AI는 속도와 반복 비용 절감에 투입된다

### Featured AI-Focused Projects 원칙

- 최대 3개만 전면 배치
- 우선 구성:
  VR 로봇 원격 조종 시스템
  너스탤지아
  DXCenter + Unity Editor Tooling
- 카드 구조는 통일:
  프로젝트명 / 한 줄 정의 / 문제 / 왜 AI가 필요했는가 / 어떻게 적용했는가 / 어떤 이슈가 있었는가 / 어떻게 해결했는가 / 결과

### Foundation Projects 원칙

- "예전 프로젝트"가 아니라 "현재 방식을 가능하게 한 기반"
- 영역 기준으로 묶기:
  VR/AR 안전교육 시뮬레이션
  대용량 BIM 시각화와 설계 자동화
  설계/저작 도구와 운영 자동화
  교육용 게임 및 서비스형 콘텐츠
- 밀도는 낮추되 신뢰는 유지

### Working Style 원칙

- 기술 스택 나열 금지
- 다음 세 문장 구조를 중심으로 작성:
  도메인과 제약을 먼저 정리한다
  반복 작업과 구조적 병목을 찾는다
  AI는 속도를 높이는 도구로 쓰되 구조와 검증은 사람이 잡는다

## 카피라이팅 방향

- 과장형 수식어보다 객관적 설명 우선
- 금지에 가까운 표현:
  AI Agent Built
  자작 framework
  단독 구현
  멀티 페르소나
- 대체 표현:
  AI 협업 기반으로 설계·구현
  역할 분리형 AI 협업 방식
  반복 가능한 개발 방식과 도구 체계 정리
  전체 구조 설계 및 핵심 구현 주도
  운영 가능한 구조로 정리
- 문장 톤:
  짧고 분명하게
  한 문장에 한 주장
  숫자는 검증 가능한 것만 사용

## 대표 프로젝트 재선정 기준

- AI가 실제 문제 해결 과정에서 구조적으로 쓰였는가
- 결과가 동작하는 시스템 또는 운영 가능한 형태로 이어졌는가
- 사용자의 핵심 포지셔닝과 직접 연결되는가
- 서로 다른 문제 축을 대표하는가

권장 조합:

1. VR 로봇 원격 조종
   XR + 로봇 + 실시간 제어 + 안전 검증
2. 너스탤지아
   AI 파이프라인 + XR 경험 설계 + 운영 구조
3. DXCenter + Unity Editor Tooling
   산업용 저작도구 + 반복 작업 자동화 + AI 협업 프로세스

## 기반 경력 섹션의 역할

- 현재 AI 활용 능력이 갑자기 생긴 것이 아님을 보여 줌
- 산업 납품, 실시간 시스템, 성능 최적화, 하드웨어 연동, 도메인 이해를 축적해 왔다는 증거 제공
- 홈과 프로젝트 페이지에서 압축 소개
- 리줌에서 상세 흐름으로 확장

## 반응형 구조 방향

- 모바일:
  한 번에 한 주장, 한 카드 안에 과도한 텍스트 금지
- 데스크톱:
  비교와 구조가 보이도록 2열 또는 3열 사용
- 섹션 순서와 의미는 화면 크기와 무관하게 동일하게 유지
- 배지와 태그보다 본문 구조를 우선

## 구현 계획

   홈, 프로젝트, 어바웃, 리줌의 title/description 재작성
6. 반응형 점검
   모바일 메뉴, 카드 높이, 텍스트 줄바꿈, CTA 일관성 확인

   변경 내용과 의도를 기록

## 구현 시 주의사항

- 기존 case study와 자산은 폐기하지 않는다.
- 없는 성과나 숫자를 추가하지 않는다.
- 프로젝트를 더 많아 보이게 만드는 대신 더 또렷하게 보이게 만든다.
- 디자인 실험보다 채용 설득력을 우선한다.
