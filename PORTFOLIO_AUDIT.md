# PORTFOLIO_AUDIT

## 분석 범위

- 라우팅: 정적 HTML 직접 링크 구조
- 페이지 구성: `index.html`, `projects.html`, `about.html`, `resume.html`, `cover-letter.html`, 4개 case study
- 공통 컴포넌트: 상단 내비게이션, 모바일 오버레이 메뉴, 푸터, Connect CTA
- 데이터 소스: 별도 JSON/API 없음, 각 HTML 파일에 콘텐츠가 직접 하드코딩됨
- 프로젝트 노출 구조: 홈 1개 featured, 프로젝트 페이지 다수 카드, resume 타임라인, cover letter 본문 반복
- 진단 항목: 카피, 정보구조, 레이아웃, 시각적 위계, 반응형 구조

## 현재 사이트의 강점

- 산업용 Unity 실시간 시스템 경력이 실제 프로젝트 단위로 풍부하게 드러난다.
- XR, BIM, Editor Tooling, 로봇 연동, 하드웨어 제어, 대용량 데이터 처리까지 범위가 넓다.
- 단순 개인 작업물이 아니라 납품, 운영, 전시, 상용/실환경 적용 경험이 있다는 점이 강하다.
- VR 로봇, DXCenter, GenWorld, Orchestration 등 최근 작업이 있어 현재성도 확보되어 있다.
- GitHub, 영상, 외부 기사, 시연 링크가 있어 신뢰 보강 자료가 충분하다.
- 모바일 메뉴와 기본 반응형 토큰이 이미 정리되어 있어 구조 개편에 필요한 기반은 있다.

## 현재 사이트의 문제점

- 첫 인상이 "산업용 Unity 개발자"보다 "AI Agent 브랜딩 개발자"에 더 가깝다.
- 홈의 핵심 구조가 채용 설득형이 아니라 브랜딩 랜딩형으로 설계되어 있다.
- 같은 메시지가 홈, 어바웃, 리줌, 커버레터에서 반복되며 페이지별 역할이 분리되지 않는다.
- 프로젝트 페이지가 강한 프로젝트와 보조 프로젝트를 같은 화면 문법으로 많이 나열해 우선순위가 흐려진다.
- "AI Agent 활용", "멀티 페르소나", "자작 framework", "단독 구현" 같은 표현이 자주 반복되어 기술적 사실보다 자기 주장처럼 읽힐 위험이 있다.
- AI를 왜 썼고, 어디에 썼고, 어떤 제약을 통제했는지보다 "AI를 썼다"는 표면 정보가 앞선다.
- 기존 산업 프로젝트들이 현재 역량의 기반으로 연결되지 않고 과거 실적 목록처럼 보이는 구간이 있다.
- 다크/글로우/그라디언트 중심 톤이 채용 친화적 읽기 경험보다는 강한 스타일 인상에 더 기여한다.
- CTA가 명확한 행동 유도보다 전반적 브랜딩 동선의 일부처럼 배치되어 있다.
- 대표 프로젝트 설명이 기술 나열형인 곳이 많아 문제 해결 구조가 한눈에 보이지 않는다.

## 메시지가 퍼지는 이유

1. 사이트 전체에서 "나는 누구인가"와 "어떻게 일하는가"와 "무엇을 했는가"가 한 페이지 안에서 동시에 경쟁한다.
2. 홈에서 3 Pillars가 정체성, 수상, AI 활용을 병렬로 보여 주어 핵심 주장을 분산시킨다.
3. AI 관련 표현이 거의 모든 페이지 상단과 카드에 반복되어 차별점이 아니라 전면 레이블처럼 보인다.
4. Foundation 역할을 해야 할 기존 산업 프로젝트가 현재 AI 활용 방식과의 인과관계로 충분히 재배치되지 않았다.
5. 페이지별 역할 구분이 약해 사용자가 홈에서 읽은 내용을 어바웃과 리줌에서 다시 비슷하게 읽게 된다.

## 유지할 요소

- 정적 HTML + Tailwind CDN + 현재 라우팅 구조
- 모바일 메뉴 패턴과 반응형 토큰
- VR 로봇, DXCenter, 너스탤지아, BIM, 솔트웍스 VR 등 기존 핵심 사례
- 외부 검증 요소: 엑스포, 장관표창, BIFAN, 납품 이력
- 시연 링크, GitHub 링크, 이미지 자산
- case study 페이지 자산과 cross-reference 체계

## 줄이거나 제거할 요소

- "AI Agent Built", "자작 framework", "단독 구현", "멀티 페르소나"의 전면 반복 노출
- 홈의 과한 브랜딩성 문구와 실적 배지 남발
- 정보 계층 없이 병렬로 나열된 기술 태그 과다 노출
- 다크 배경 위 글로우/유리 효과 중심의 과한 시각 장치
- 페이지 간 중복되는 자기소개 문단
- 대표 프로젝트보다 주변 프로젝트가 더 많이 보이는 배열

## 가장 먼저 고쳐야 할 우선순위 10개

1. 홈 Hero의 첫 문장을 "산업용 Unity 실시간 시스템 개발자" 중심으로 다시 정의
2. 홈에 `How I Actually Use AI` 섹션을 신설하고 엔지니어링 워크플로우처럼 구조화
3. 홈 Featured Projects를 최대 3개로 축소하고 문제 해결 구조로 재작성
4. 홈 Foundation Projects를 신뢰 기반 섹션으로 별도 분리
5. 홈 Working Style을 기술 스택이 아닌 문제 해결 방식 중심으로 재구성
6. Projects 페이지를 "대표 3개 + 기반 경력" 구조로 재편
7. About 페이지를 브랜딩 소개에서 일하는 방식과 판단 기준 중심으로 전환
8. Resume 페이지를 기술 나열보다 성장 흐름이 보이는 순서로 재정렬
9. 전 페이지 상단 메타 설명과 헤드라인에서 과장성 AI 표현을 줄이고 객관적 문장으로 교체
10. 전체 시각 톤을 밝고 차분한 editorial 방식으로 정리해 가독성과 채용 친화성을 높이기

## 구조 진단 메모

### 라우팅

- 모든 페이지는 정적 HTML 간 직접 링크
- 홈을 허브로 하고 프로젝트, 어바웃, 리줌, 커버레터, 케이스 스터디로 이동
- 데이터 레이어 없이 각 페이지가 독립적으로 카피를 관리

### 공통 컴포넌트

- 내비게이션과 모바일 오버레이 메뉴는 일관성이 높음
- Connect CTA는 페이지별 위치 차이가 있으나 기능은 동일
- 푸터는 단순하며 유지 가능

### 프로젝트 카드 구조

- 현재 `projects.html`은 시각적으로는 강하지만 카드 구조가 제각각이다.
- 일부는 강한 카드, 일부는 링크 카드, 일부는 OSS/R&D 카드로 섞여 있다.
- 대표 프로젝트와 기반 프로젝트가 같은 레벨로 보여 전략적 우선순위가 약하다.

### CTA 구조

- 홈 CTA는 `프로젝트 보기`, `이력서 보기`로 나쁘지 않지만 메시지와 더 직접 연결될 필요가 있다.
- 최종 CTA는 "연락하라"보다 "어떤 문제를 함께 풀 수 있는가"가 보여야 더 설득력 있다.

### 반응형 구조

- 기본 토큰은 안정적이다.
- 다만 카드 내부 정보가 많아 모바일에서 읽기 부담이 생기는 구간이 있다.
- 이번 개편에서는 정보를 줄이기보다 구조를 단계화해서 모바일에서도 같은 의미가 유지되게 해야 한다.
