The Kernel

Ops-Cure — channel-native state/event kernel

A channel is not just a message pipe. It is a stateful space with actors and events. Ops-Cure makes that treatment explicit: behaviors plug in, transports adapt, runtimes execute.

Channel-native kernel
Space · Actor · Event · Behavior

The kernel stays generic. Domain semantics live inside behaviors. Machine-specific execution stays in runtimes.

Core vocabulary

Space. A stateful room — a Discord channel, thread, or session that owns its own state over time.

Actor. A participant — human, AI worker, bot, or system.

Event. A recorded fact inside a space. Transport-agnostic, persistable, replayable.

Behavior. Domain logic that interprets events and maintains domain state on top of the kernel.

Why three behaviors matters

One behavior is a feature. Three fundamentally different behaviors — structured workflow, remote execution with approval, free-form dialogue — running on the same kernel is a claim that the abstraction is real. Each behavior validates a different axis: orchestration tests the state/task model, remote_codex tests canonical evidence under distributed runtime, chat tests room generality without task semantics.

Production SaaS · Remote Agent Runtime

DeskRelay — browser/mobile control plane for local agent execution

A SaaS platform that lets a user pair multiple PCs, choose one from the browser or Android app, and drive a local agent CLI through a PC connector. The first adapter targets Claude Code, but the product shape is a local-first relay/control plane rather than a Claude-only wrapper.

What I built

Product architecture. A monorepo spanning shared contracts, behavior SDK, PC connector daemon/CLI, Cloudflare Workers backend, SolidJS frontend, and Capacitor Android packaging.

Device lifecycle. Pairing code flow, device identity, diagnostics, per-device settings, and unpair cleanup that closes the live relay and clears local browser/device state.

Runtime boundary. The site does not store chat transcripts or code. In alpha it relays request bodies through a Durable Object; E2E/WebRTC hardening is documented as the next trust-model step.

Release surface. winget/Homebrew templates, Android listing copy, localization source, and operator runbooks sit beside the code so the product can be shipped rather than only demoed.
