왜 behavior 세 개가 중요한가

역할

커널(Space/Actor/Event/Behavior 추상화), Discord transport, behavior 플러그인 레지스트리, canonical DB 설계·구현. 각 behavior 계약 설계와 orchestration, remote_codex, chat behavior 전반을 소유.

문제

에이전트 프레임워크는 하나의 워크플로에 굳어지는 경향이 있음. 원격 실행, AI 간 대화, 인시던트 방 등 새로운 상호작용 형태가 생길 때마다 코드베이스가 분기되고 공통 상태 모델이 썩는다.

접근

채널 자체를 추상화로 취급. 커널은 Space/Actor/Event/Behavior 만 안다. 도메인 로직은 behavior 플러그인, runtime 은 adapter. 특수 분기 없이 전혀 다른 behavior 를 같은 커널이 수용.

결과

프로덕션 또는 그 근처의 behavior 3개 — orchestration(프로덕션), remote_codex(계약 안정화 중), chat(라이브). 각각이 커널 범용성의 다른 축을 검증.

§ 01 · 커널

Space · Actor · Event · Behavior

Ops-Cure의 핵심은 커널이 약속하는 범용 어휘. 커널은 태스크, handoff, Codex, Discord 스레드 어느 것도 구체적으로 알지 않는다.

커널 구조
Kernel · Behaviors · Transports · Runtimes

커널은 범용. 나머지가 맞춘다.

KernelSpace / Actor / Event / Behavior
Behaviorsorchestration · remote_codex · chat · ops · game
TransportDiscord 봇, 스레드 매니저, presenter
Runtimespc_launcher · remote_executor · chat_participant
hubbehavior는 설치형 플러그인, 분기 아님.
어휘

Space. 상태가 있는 방 — Discord 채널, 스레드, 세션 — 이 시간 축으로 상태를 누적. 프로세스 재시작을 넘어 지속.

Actor. 공간에 붙은 참여자: 사람, AI 워커, 봇, 시스템. Identity 와 permission 은 있지만 behavior 는 없다.

Event. 공간 안에 기록된 사실. Transport 독립적, 저장/재생 가능. 모든 behavior 는 이벤트를 쓰고 읽는다.

Behavior. 커널 위에 얹힌 도메인 로직. 이벤트를 해석하고 자체 도메인 상태(태스크, 메시지, evidence — 도메인이 요구하는 것)를 유지하며, 범용 Space/Actor/Event view 로 투영.

커널은 orchestration 태스크가 무엇인지 모른다. 다만 "어떤 behavior 가 어떤 공간에 이벤트를 썼고, 어떤 actor 가 참여했다" 만 안다.

§ 02 · Behavior — orchestration

Discord-native 멀티에이전트 워크플로

프로덕션 behavior. planner · curator · coder · verifier · reviewer 가 canonical task 와 handoff 위에 협업. 각 Discord 스레드가 한 워크플로 세션이 된다.

forumDiscord 스레드

스레드가 세션 키. 워크스페이스 경로, 주요 파일, 라우팅이 같은 스레드 안에서 보인다. 평문은 기본적으로 planner 로 라우팅, @coder / @reviewer 태그로 override.

chatArtifact 리포트

운영자용 요약 + 로컬 워크스페이스 경로 + 주요 artifact(RUN_LOGS, REPORT, HANDOFFS). 실행 로그 원본을 Discord 에 그대로 쏟아내지 않는다.

orchestration 이 소유하는 것

Canonical task. planner 가 분해한 태스크 모델, self-claim 스케줄링, 역할 간 명시적 handoff.

Workflow projection. 각 워커가 마크다운 artifact — CURRENT_STATE.md, TASK_BOARD.md, HANDOFFS.md, REPORT.md — 를 유지, bridge 는 이를 evidence 로 읽는다.

Adapter 가드레일. 고정된 codex, claude, mock adapter 가 로컬 CLI 를 감싼다. Discord 입력이 raw shell 이 되지 않는다.

Drift 모니터. Heartbeat 에 artifact 스냅샷을 실어, 살아있어 보이지만 공유 상태를 업데이트하지 않는 워커를 플래그.

히스토리

orchestration behavior 는 초기에 TypeScript 기반 독립 프레임워크에서 출발했다 — 역할 기반 collaboration, 리뷰 규칙, launcher 대시보드를 묶은 사고 실험. 그 프로토타입은 collaboration primitive 를 검증했지만, canonical state 없이는 실제 운영자를 견디지 못했다.

지금의 orchestration 은 그 프로덕션 후계: 같은 collaboration primitive 를 커널 위 플러그인으로 재구현. 이전 TS 프레임은 이 behavior 안으로 흡수, 은퇴.

Runtime: pc_launcher/ (Windows) — manifest 등록, 세션 claim, agent 당 워커 프로세스 1개, CLI adapter 실행.

Status · Production Runtime · pc_launcher Agents · planner · curator · coder · verifier · reviewer
§ 03 · Behavior — remote_codex

브라우저 기반 원격 실행

두 번째 behavior. 제어면이 Discord 가 아니라 브라우저 사이트. canonical state 는 bridge 가 소유하고 브라우저는 렌더링만.

remote_codex 흐름
Browser → Bridge → Executor

canonical task · evidence · approval 은 bridge 안. 브라우저는 projection, executor 는 adapter.

Browsercodex-remote 사이트 — typed read model 소비
Bridgecanonical task / evidence / approval / stall 상태
Runtime대상 PC 의 remote_executor 커넥터
Discordcoordination mirror, canonical 아님
swap_horiz브라우저가 태스크 상태의 소유자가 되면 안 된다.
Ownership 규칙

behavior 가 소유: machine + canonical thread binding, remote work object 와 assignment state, heartbeat + evidence state, approval/interrupt/stall state, 브라우저용 typed read model, 범용 Space/Actor/Event 로의 projection.

behavior 가 소유하지 않음: 브라우저 레이아웃, Codex 프로세스 제어 세부, Windows UTF-8, Discord 메시지 에티켓, optimistic 렌더링. 이건 adapter 쪽.

왜 이 경계가 중요한가: 사이트 전용 상태는 유혹적이다. 브라우저가 source of truth 가 되는 순간 다른 transport(모바일 앱, CLI, 또 다른 사이트) 를 붙일 수 없다 — fork 만 남는다. Bridge 소유 계약이 behavior 를 transport 독립으로 유지한다.

현재 상태 (솔직히)

패키지는 현재 product-layer 의 remote task 서비스를 감싼 얇은 behavior 스캐폴드. 라이브 wiring (kernel binding, Discord binding) 은 의도적으로 아직 미활성 — 계약이 안정화되는 중이다. 안정화되면 현재 bridge 서비스와 브라우저 전용 상태에 흩어져 있는 remote task 로직이 이 패키지로 통합된다.

이걸 포트폴리오에 넣는 이유는 설계 가 포인트이기 때문. behavior / 브라우저 / Discord 미러 / Windows executor 간의 ownership 분할은 "제어면이 바뀌면 어떻게 되는가?" 에 대한 의도적 답변이다.

Status · Scaffold · 계약 안정화 중 Runtime · remote_executor 파트너 사이트 · codex-remote (browser)
§ 04 · Behavior — chat

Codex 끼리의 대화방

범용성 점검. 태스크도, handoff 도, evidence 도 없는 behavior — 방, 참여자, 메시지뿐. 여기서 커널이 성립하면 실체다.

chat 흐름
Thread → Participants → Messages

워크플로 없음. Actor 와 Event 만 있는 방.

Roomchat space 로서의 Discord 스레드
ParticipantsAI 에이전트 또는 사람, 방 단위 등록
Messagessubmit · delta fetch · heartbeat
Runtime각 PC 의 chat_participant 커넥터
forum태스크 없음. Handoff 없음. 대화만.
chat 이 진짜 검증인 이유

커널 어휘가 은연중 orchestration(태스크, handoff, reviewer) 에 맞춰 설계되어 있었다면, 그런 개념이 없는 behavior 를 추가할 때 추상화의 구멍이 드러난다. chat 은 그 개념이 전혀 없다. 방, actor, 메시지만 있으면 된다.

chat 은 orchestration 과 정확히 같은 Space/Actor/Event 어휘 위에서 돌아간다. 커널이 "태스크 모양" 으로 편향되지 않았다는 구체적 증거.

Runtime: chat_participant 커넥터 — 가벼운 PC 측 프로세스, 방 delta 를 구독, 메시지를 post, 참여자 heartbeat 유지.

유스케이스: Codex 간 대화, 사람-AI 대화방, workflow 엔진 없이 PC 간 대화.

Status · Live Runtime · chat_participant 목적 · 커널 범용성 증명
§ 05 · 왜 behavior 세 개가 중요한가

behavior 한 개는 기능. 세 개는 주장.

orchestration

상태/태스크 모델을 검증. 커널이 구조화된 워크플로를 특수 분기 없이 수용.

remote_codex

분산 runtime 아래의 canonical evidence 를 검증. 제어면이 브라우저로 옮겨가도 bridge 가 single source of truth 로 남는다.

chat

태스크 시맨틱 없는 방의 범용성을 검증. 순수 대화에서도 커널이 성립하면, 커널이 orchestration 에 몰래 편향되지 않았다는 증거.

중요한 축은 behavior 수가 아니라 behavior 의 폭. 3개가 의미 있는 건, 각각이 커널의 다른 부분을 압박하기 때문이다. orchestration 모양의 behavior 를 다섯 개 더 만들었다면 그건 feature farm 이지 플랫폼이 아니다.
