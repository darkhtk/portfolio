Why three behaviors matter

Role

Designed and implemented the kernel (Space/Actor/Event/Behavior abstraction), the Discord transport, behavior plugin registry, and the canonical DB. Designed each behavior contract and owns the orchestration, remote_codex, and chat behaviors.

Problem

Agent frameworks tend to solidify around one workflow. Any new interaction shape — remote execution, AI-to-AI chat, incident rooms — forks the codebase. The shared state model rots.

Approach

Treat the channel itself as the abstraction. A kernel that only knows Space/Actor/Event/Behavior. Domain logic lives in plugin behaviors. Runtimes are adapters. The same kernel hosts very different behaviors without special-casing.

Outcome

Three behaviors in production or near-production — orchestration (mature), remote_codex (contracts stabilizing), chat (live). Each validates a different axis of the kernel's generality.

§ 01 · The Kernel

Space · Actor · Event · Behavior

The core of Ops-Cure is a generic vocabulary the kernel commits to. Nothing in the kernel knows about tasks, handoffs, Codex, or Discord threads specifically.

Kernel layout
Kernel · Behaviors · Transports · Runtimes

The kernel is generic. Everything else adapts.

KernelSpace / Actor / Event / Behavior
Behaviorsorchestration · remote_codex · chat · ops · game
TransportDiscord bot, thread manager, presenter
Runtimespc_launcher · remote_executor · chat_participant
hubBehaviors are installable plugins, not branches.
Vocabulary

Space. A stateful room — Discord channel, thread, or session — that owns state over time. Spaces persist across process restarts.

Actor. A participant attached to a space: human, AI worker, bot, or system. Actors have identity and permissions but not behavior.

Event. A recorded fact inside a space. Events are transport-agnostic, persistable, replayable. Every behavior writes and reads events.

Behavior. Domain logic layered on top of the kernel. A behavior interprets events, maintains its own domain state (tasks, messages, evidence — whatever the domain requires), and projects back into generic Space/Actor/Event views.

The kernel does not know what an orchestration task is. It only knows that some behavior wrote an event into a space, and some actor participated.

§ 02 · Behavior — orchestration

Discord-native multi-agent workflow

The production behavior. Planner, curator, coder, verifier, reviewer collaborate on canonical tasks and handoffs. Each Discord thread becomes a workflow session.

forumDiscord thread

Thread as session key. Workspace path, key files, and routing appear inline. Plain text routes to the planner by default; @coder, @reviewer overrides.

chatArtifact report

Operator-facing summary plus local workspace path and key artifacts (RUN_LOGS, REPORT, HANDOFFS). Execution logs never dumped raw into Discord.

What orchestration owns

Canonical tasks. A task model with planner-generated decomposition, self-claim scheduling, explicit handoffs between roles.

Workflow projection. Each worker maintains markdown artifacts — CURRENT_STATE.md, TASK_BOARD.md, HANDOFFS.md, REPORT.md — that the bridge reads as evidence.

Adapter guardrails. Fixed codex, claude, mock adapters wrap local CLIs. Discord input never becomes raw shell.

Drift monitor. Heartbeats carry artifact snapshots. Workers that look alive but are not updating shared state get flagged.

History

The orchestration behavior began as a standalone TypeScript framework for role-based AI collaboration — a thought experiment in review rules, handoff contracts, and launcher dashboards. That prototype validated the collaboration primitives but couldn't survive real operators without canonical state.

The current orchestration is the production descendant: same collaboration primitives, now running as a plugin on a kernel that owns state, transport, and runtime routing. The TS framework is retired into this behavior.

Runtime: pc_launcher/ on Windows — manifest registration, session claim, one worker process per agent, CLI adapter execution.

§ 03 · Behavior — remote_codex

Browser-first remote execution

The second behavior. Instead of Discord as the control surface, a browser site drives remote Codex execution. The bridge owns canonical state. The browser only renders.

remote_codex flow
Browser → Bridge → Executor

Canonical task, evidence, and approval truth lives in the bridge. The browser is a projection; the executor is an adapter.

Browsercodex-remote site — reads typed read models
Bridgecanonical task / evidence / approval / stall state
Runtimeremote_executor connector on the target PC
Discordcoordination mirror, not canonical
swap_horizThe browser cannot become the owner of task state.
Ownership rules

The behavior owns machine + canonical thread bindings, remote work objects and assignment state, heartbeat + evidence state, approval / interrupt / stall state, typed read models for the browser, and projection into generic Space/Actor/Event.

The behavior does not own browser layout, Codex process control details, Windows UTF-8 handling, Discord message etiquette, or optimistic rendering. Those are adapters.

Why this boundary matters: site-specific state is seductive. Once the browser becomes the source of truth, another transport (mobile app, CLI, another site) cannot be added without forking. The bridge-owned contract keeps the behavior transport-independent.

Current state (honest)

The package is a thin behavior scaffold wrapping the current product-layer remote task service. Live wiring (kernel binding, Discord binding) is intentionally not enabled yet — the contracts are stabilizing. Once stable, the remote task logic folds into this package from its current spread across bridge services and browser-only state.

This is in the portfolio because the design is the point: the ownership split between behavior, browser, Discord mirror, and Windows executor is a deliberate answer to the question "what happens when the control surface changes?"

§ 04 · Behavior — chat

Codex-to-Codex dialogue rooms

The generality check. A behavior with no tasks, no handoffs, no evidence state — just rooms, participants, and messages. If the kernel holds here, it's real.

chat flow
Thread → Participants → Messages

No workflow. Just a room with actors and events.

RoomDiscord thread as chat space
ParticipantsAI agents or humans, registered per room
Messagessubmit · delta fetch · heartbeat
Runtimechat_participant connector on each PC
forumNo tasks. No handoffs. Just dialogue.
Why chat is the real test

If the kernel vocabulary were secretly shaped by orchestration (tasks, handoffs, reviewers), adding a behavior without those concepts would reveal leaky abstractions. Chat has none of that. It only needs rooms, actors, messages.

Chat runs on the same Space/Actor/Event vocabulary as orchestration does. The kernel didn't grow a "task-shaped" bias. That is the concrete evidence that the abstraction holds.

Runtime: chat_participant connector — lightweight PC-side process, subscribes to room delta, posts messages, maintains participant heartbeat.

Use case: Codex-to-Codex dialogue, human-to-AI chat room, cross-PC conversation without the overhead of a workflow engine.

§ 05 · Why three behaviors matter

One behavior is a feature. Three is a claim.

orchestration

Validates the state and task model. The kernel carries structured workflow without special-casing it.

remote_codex

Validates canonical evidence under distributed runtime. The bridge stays the single source of truth when the control surface moves to the browser.

chat

Validates room generality without task semantics. If the kernel holds for pure dialogue, it isn't quietly shaped by orchestration.

The axis that matters is breadth of behavior — not count of behaviors. Three behaviors only count because they stress different parts of the kernel. If I built five more orchestration-shaped behaviors, I would have a feature farm, not a platform.
